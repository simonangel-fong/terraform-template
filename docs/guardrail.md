# Terraform Module Guardrail

Authoring rules for every module in this repo. Keep modules small, typed, and testable.

## Versions

- Terraform: `>= 1.6`
- Pin every provider in `versions.tf` with `~>` (minor-version drift only)
- Modules **do not** configure providers â€” the caller does

## File layout

Each module is a flat directory. No subdirectories except `tests/`.

| File                 | Purpose                                                      |
| -------------------- | ------------------------------------------------------------ |
| `01_variables.tf`    | Input variables â€” required first, optional below             |
| `02_locals.tf`       | Internal variables                                           |
| `03_providers.tf`    | `terraform` block: `required_version` + `required_providers` |
| `04_outputs.tf`      | Output values; mark secrets `sensitive = true`               |
| `README.md`          | Auto-generated by `terraform-docs` â€” do not hand-edit        |
| `tests/*.tftest.hcl` | At least one plan-mode test per module                       |

- main code block:
  - `05_main.tf`: when the main block is simple.
  - `<number>_<infra_layer_name>.tf`:
    - when multiple layers required.
      - number follows infrastructure layers, the more fundamental, the lower number.
    - e.g., `05_vnet.tf`,`06_vm.tf`

## Naming

- Files, variables, outputs, locals, resource labels: `snake_case`
- Singleton resource label: `this` (e.g. `azurerm_kubernetes_cluster.this`)
- Module directory: short noun, no provider prefix (`aks`, not `azure-aks`)
- No abbreviations that aren't obvious in the cloud's own docs

## Variables

- Every variable has `type` and `description`
- Required variables: no `default`. Optional: `default` set (use `null` for "unset")
- Group related options into a single `object({ ... })` with `optional(..., <default>)` per field
- Validate inputs that have a finite domain with `validation { condition = ..., error_message = ... }`
- Never accept secrets as plain `string` without marking the variable `sensitive = true`

## Outputs

- Expose IDs, names, and anything a caller needs to wire to another module
- `sensitive = true` for kubeconfigs, connection strings, keys
- Include a `description` on every output

## Resources

- One resource per logical thing; use `count` / `for_each` for toggles and sets, not duplication
- Toggle pattern: `count = var.enable_<x> ? 1 : 0` and reference `resource.this[0].id`
- Use `dynamic` blocks for optional nested configuration driven by a variable
- Prefer `coalesce(var.x, <fallback>)` over conditional assignment
- Tag everything taggable via a `tags` variable (`map(string)`, default `{}`)

## State & backends

- Modules **never** declare a `backend` block â€” that belongs to the root/caller
- Never commit `.terraform/`, `*.tfstate`, `*.tfvars` with secrets, or `.terraform.lock.hcl` from a module subdirectory (lockfile lives with the root config)

## Testing & CI

- `tests/*.tftest.hcl` â€” at least one `command = plan` test exercising required variables
- Pre-commit: `terraform_fmt`, `terraform_validate`, `terraform_docs`
- CI runs `terraform fmt -check -recursive`, `terraform init -backend=false`, `validate`, `test` per module

## Consumption

Pinned by tag, never by branch:

```hcl
source = "git::https://github.com/simonangel-fong/terraform-template.git//azure/aks?ref=v1.0.0"
```

## Best practices

- **Smallest surface that works.** Don't add inputs for hypothetical needs â€” add them when a caller asks
- **Sane defaults over required inputs.** If 80% of callers pass the same value, default it
- **Fail at plan, not apply.** Use `validation` blocks and `precondition`/`postcondition` to surface bad inputs early
- **Idempotent and re-runnable.** No `local-exec` for state-changing work; no `time_sleep` as a synchronization primitive
- **Version every breaking change.** Tag releases; bump major on input/output renames or removed resources
- **Document the contract, not the code.** READMEs describe inputs/outputs and assumptions; the code shows how
